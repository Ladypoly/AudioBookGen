APPENDIX A: ENGINE PROMPTING CONTRACTS (authoritative)
==================================================================

Local engine reality (all confirmed running locally on this machine):
- LLM: huihui_ai/gemma-4-abliterated:26b in Ollama
- TTS: Higgs Audio v3 (boson-ai/higgs-audio), driven via the
  TTS-Audio-Suite custom nodes in ComfyUI
  (https://github.com/diodiogod/TTS-Audio-Suite)
- SFX/music: Stable Audio 3 in ComfyUI
- Covers: Ideogram 4 (and Z-Image Turbo) in ComfyUI

Core design rule:
The LLM produces ONE structured, enum-locked output per line/cue.
Dedicated builders translate that single structure into:
  (a) Higgs control-token text   -> TagBuilder
  (b) Ideogram JSON caption      -> CoverCaptionBuilder
The LLM never emits engine-specific syntax. The builders are the
only code that knows engine syntax. Validate LLM output against the
enums below BEFORE it reaches a builder.

------------------------------------------------------------------
A.1  HIGGS AUDIO v3 — control tokens (verified from model card
     bosonai/higgs-audio-v3-tts-4b)
------------------------------------------------------------------
Tag syntax: <|category:value|>  (insertable mid-utterance)
Placement: delivery tokens (emotion, style, prosody speed/pitch)
go at the START of the turn and shape the whole turn. Pauses, pitch
shifts, and sfx tokens go INLINE where they occur. Each sfx token
must be immediately followed by matching onomatopoeia.

Example:
<|emotion:fear|><|style:whispering|><|prosody:speed_slow|>Was war das?<|prosody:pause|>

ALLOWED ENUMS — the LLM schema MUST constrain to exactly these.
TagBuilder drops or maps anything off-list; validation rejects
unknown values rather than passing them through.

emotion (21):
  elation, amusement, enthusiasm, determination, pride,
  contentment, affection, relief, contemplation, confusion,
  surprise, awe, longing, arousal, anger, fear, disgust,
  bitterness, sadness, shame, helplessness

style (3):
  singing, shouting, whispering

prosody (10):
  speed_very_slow, speed_slow, speed_fast, speed_very_fast,
  pitch_low, pitch_high, pause, long_pause,
  expressive_high, expressive_low

sfx (9) — each REQUIRES onomatopoeia immediately after:
  cough, laughter, crying, screaming, burping, humming,
  sigh, sniff, sneeze
  e.g. <|sfx:laughter|>Haha   <|sfx:sigh|>Uh

Note: 'arousal' emotion exists in the model but should be filtered
out of the LLM's allowed set for this product unless explicitly
relevant; default-exclude it in the schema enum.

Reference-audio cloning:
- format: references: [{ audio_path, text }]
- the reference TRANSCRIPT materially improves fidelity and is
  REQUIRED in practice. The voice library (screen 6) MUST store a
  ref_text alongside every sample. Add column voices.ref_text.
- multi-speaker = supply one reference per speaker.

TagBuilder contract:
  build_higgs_text(delivery, text) -> str
  - prepend turn-level tokens in fixed order:
    emotion, style, prosody(speed), prosody(pitch), expressive
  - inline-insert pauses and sfx at their marked offsets
  - assert every token is in the allowed enum (defensive)

Map your existing schema (PLAN line ~290) delivery fields:
  emotion          -> <|emotion:VALUE|>     (enum above)
  style            -> <|style:VALUE|>        (enum above)
  prosody[]        -> <|prosody:VALUE|>      (enum above)
  nonverbal[]      -> <|sfx:VALUE|>onomatopoeia
  pre/post_pause_ms-> handled by MIXER, not tokens (keep ms in mix
                      timeline; only use <|prosody:pause|> for
                      intra-line beats)

------------------------------------------------------------------
A.2  IDEOGRAM 4 — JSON caption prompting (cover consistency)
------------------------------------------------------------------
Ideogram 4 is trained EXCLUSIVELY on structured JSON captions that
exhaustively describe the image. Plain-text prompts work but are
strictly worse and drift between chapters -> inconsistent series.
Therefore cover_service builds a JSON caption object, never a
prose string. Magic Prompt MUST be OFF (it re-expands text per call
via an LLM and breaks series consistency — we own the JSON).

VERIFIED SCHEMA (from ideogram4 docs/prompting.md). Three top-level
fields, STRICT key order (CaptionVerifier warns on violations):

  high_level_description : string  (1-2 sentence summary of image)
  style_description      : object  (exactly one of photo|art_style)
      key order (non-photo): aesthetics, lighting, medium,
                             art_style, color_palette
      key order (photo):     aesthetics, lighting, photo, medium,
                             color_palette
      color_palette : list of up to 16 UPPERCASE #RRGGBB hex
  compositional_deconstruction : object (REQUIRED)
      background : string (environment)
      elements   : list of element objects
          obj  element key order: type, bbox, desc, color_palette
          text element key order: type, bbox, text, desc, color_palette
          type  : "obj" | "text"
          bbox  : [y_min, x_min, y_max, x_max], 0-1000 normalized
          color_palette : up to 5 hex per element

Serialize with json.dumps(..., separators=(",",":"),
ensure_ascii=False). Use art_style (not photo) for illustrated
covers. There is no "aspect_ratio" field in the caption — aspect
ratio is a generation param (multiple-of-16 dims), set on the
ComfyUI node, not in the JSON.

SERIES CONSISTENCY MECHANISM:
Hold these fields IDENTICAL across every chapter cover, derived
once from the Style Bible. Vary ONLY the per-chapter motif.

  FIXED (from Style Bible, all chapters):
    style_description.*    (aesthetics, lighting, medium, art_style)
    style_description.color_palette   e.g.
        ["#1A2238","#9DAAF2","#F4DB7D"]   (uppercase #RRGGBB)
    compositional_deconstruction.background   (same setting feel)
    the title/number TEXT element's bbox + style (if typography on)

  VARIES (per chapter, spoiler-safe abstract motif only):
    high_level_description           (abstract, no plot)
    the obj elements' desc           (abstract motif, no characters
                                      unless scene-detail mode is on)

CoverCaptionBuilder contract:
  build_caption(style_bible, chapter_motif, opts) -> dict
  emitting (key order matters):
    {
      "high_level_description": "<abstract chapter motif>",   # VARIES
      "style_description": {                                   # FIXED
        "aesthetics": "muted gouache, grainy paper texture",
        "lighting":   "soft diffuse light",
        "medium":     "illustration",
        "art_style":  "painterly storybook",
        "color_palette": ["#1A2238","#9DAAF2","#F4DB7D"]
      },
      "compositional_deconstruction": {
        "background": "<fixed setting feel from bible>",       # FIXED
        "elements": [
          {"type":"obj","bbox":[300,200,800,800],
           "desc":"<abstract chapter motif object>"},          # VARIES
          {"type":"text","bbox":[850,100,950,900],
           "text":"Kapitel 3","desc":"title text, serif"}       # FIXED pos
        ]
      }
    }
The master anchor cover uses the same builder with a series-level
motif; you MAY also pass it as an image reference, but the JSON
(fixed style_description + background) is the primary anchor.

Cache key for a cover = hash(serialized caption JSON + backend +
workflow template version). Changing the bible invalidates all
covers. Run the doc's CaptionVerifier on every built caption.

------------------------------------------------------------------
A.3  SPOILER-SAFE CHARACTER COLLECTION (map-reduce over whole book)
------------------------------------------------------------------
The LLM runs over the WHOLE PDF, chunked, and must collect every
character with a short description and a VOICE description while
leaking minimal plot. Spoiler-safety is enforced at the PROMPT
CONTRACT level (output is constructed spoiler-light at the source),
not only by hiding things in the UI.

MAP pass (per chunk) — structure_scan + character mentions:
  For each character mention in the chunk, emit ONLY:
    name_surface (as written), gender_guess(enum), age_band(enum),
    observable vocal/physical traits, mention_count_in_chunk
  HARD PROMPT RULES (forbidden):
    - no plot events, deaths, twists, betrayals, outcomes

==================================================================
APPENDIX B: STABLE AUDIO 3 — SFX / AMBIENCE / MUSIC GENERATION
==================================================================

The plan currently treats "SFX" as one thing. It is THREE distinct
asset classes with different generation, length, and placement
rules. Model them separately in schema, cache, and mixer.

------------------------------------------------------------------
B.1  THREE AUDIO ASSET CLASSES
------------------------------------------------------------------
1) ONE-SHOT SFX  (transient events)
   examples: door, glass break, footstep, paper, creak, thunder hit
   - short (~0.5–5 s), generated at natural length
   - placed at an ABSOLUTE timestamp relative to a line
   - NOT looped. NOT ducked (they are foreground events)
   - one cue = one WAV, trimmed to the transient

2) BACKGROUND SFX / AMBIENCE BED  (continuous texture)
   examples: harbour, rain, wind, forest, crowd murmur, room tone,
             fire, water, market, ship deck
   - must fill an ARBITRARY scene duration (could be 20 s or 6 min)
   - generate ~180 s (Stable Audio 3 max single-gen length — verify
     against your ComfyUI workflow; if capped lower, generate at the
     cap), then convert to a SEAMLESS LOOP (B.2) and tile to fill
   - sits LOW in the mix, DUCKED under dialogue (sidechain)
   - one ambience type = one looped WAV, reused across the scene

3) MUSIC CUE  (atmospheric transition)
   examples: chapter intro, scene transition sting, tension pad
   - typically 3–10 s, one-shot, NOT looped (unless an evolving pad
     explicitly intended as a bed — then treat like class 2)
   - placed at scene/chapter boundaries, low level, ducked

LLM Pass-4 cue schema MUST tag which class a cue is:
  cue_class: one_shot_sfx | ambience_bed | music_cue
The mixer branches on cue_class. Ducking required = true for
ambience_bed and music_cue, false for one_shot_sfx (default).

------------------------------------------------------------------
B.2  SEAMLESS LOOP PIPELINE (for ambience_bed)
------------------------------------------------------------------
Goal: turn a ~180 s generated clip into a loop with no audible
seam at the wrap point, then tile it to the scene length.

Steps (Python, in mixer_service or a dedicated loop_util):
  1. Generate ambience at full length (~180 s), 44.1 kHz, no vocals.
  2. Trim head/tail silence and any DC offset (high-pass ~20 Hz).
  3. SEAMLESS-LOOP via equal-power crossfade fold:
       - choose crossfade length X (e.g. 2–4 s)
       - take the last X seconds (tail) and the first X seconds
         (head); crossfade tail INTO head using an equal-power
         (constant-power, sin/cos) curve, NOT linear — linear
         crossfades dip in perceived loudness at the seam
       - the body between the folded regions is the loop core;
         result length = original_len - X
       - the join is now continuous: end flows into start
  4. Validate: RMS continuity across the wrap, no click (check for
     sample discontinuity / large delta at the boundary).
  5. Tile/repeat the loop to >= scene_duration, then HARD-CUT to
     exact scene length and apply a short fade-out (and fade-in)
     so the bed enters/exits cleanly.

PROMPTING RULES for loopable ambience (critical):
  - prompt for CONTINUOUS, STEADY texture: "constant", "steady",
    "even", "no sudden changes", "seamless ambient bed"
  - AVOID salient one-time transients inside the bed prompt. A
    single distinct seagull cry or door slam baked into a 180 s
    harbour bed becomes obviously PERIODIC once looped (you hear
    it every loop period). Keep distinct events OUT of the bed —
    generate them as separate class-1 one-shots layered on top at
    irregular timestamps.
  - explicitly state "no vocals, no music, no speech"
  - good harbour bed prompt: "steady harbour ambience, gentle water
    lapping, distant indistinct crowd murmur, faint rigging creak,
    even continuous texture, no distinct events, no vocals, field
    recording"
  - the distinct seagull/bell/horn -> separate one_shot_sfx cues

Cache key for an ambience loop:
  hash(prompt, target_gen_len, crossfade_len, sa_workflow_version)
The looped artifact is cached; the per-scene tiled/cut version is
derived cheaply at mix time and need not be cached.

------------------------------------------------------------------
